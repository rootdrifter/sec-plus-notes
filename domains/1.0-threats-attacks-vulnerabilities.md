# 1.0 — Threats, Attacks & Vulnerabilities

## 1. Threat actors and attributes

| Actor | Motivation | Sophistication / resources | Notes |
|-------|-----------|----------------------------|-------|
| **Script kiddie** | Thrill, notoriety | Low — uses others' tools | Opportunistic |
| **Hacktivist** | Ideology/politics | Low–medium | Defacement, DoS, leaks |
| **Insider** | Revenge, money, negligence | Variable — has legitimate access | Hardest to detect; privilege abuse |
| **Organised crime** | Financial | Medium–high, well-funded | Ransomware, fraud, data theft |
| **Nation-state / APT** | Espionage, sabotage, strategic | Very high; 0-days; long dwell time | "Advanced Persistent Threat" — stealth + persistence |
| **Competitor** | Commercial advantage | Variable | Industrial espionage |

**Attributes that matter:** internal vs external, level of sophistication, resources/funding, and
**intent**. **APT** = the key term: *advanced* (custom tooling, 0-days), *persistent* (stays for
months), *threat* (organised, funded — usually nation-state).

**Threat intelligence sources:** OSINT, ISACs, vendor feeds, the **dark web**, **STIX/TAXII**
(structured threat-intel exchange formats), and frameworks like **MITRE ATT&CK** (adversary TTPs)
and the **Cyber Kill Chain**.

> Portfolio link: the [mirage](../../mirage) research models the *psychology* attackers exploit
> (urgency/authority/trust), one layer above ATT&CK's technique catalogue.

## 2. Attack types

### 2.1 Social engineering

The human attack surface. Underlying **principles** (Cialdini-style): **authority, intimidation,
consensus/social proof, scarcity, urgency, familiarity/liking, trust**.

- **Phishing** (email), **spear phishing** (targeted), **whaling** (executives), **vishing**
  (voice), **smishing** (SMS), **pharming** (DNS/hosts redirection to fake site), **watering hole**
  (compromise a site the target visits).
- **Pretexting, tailgating/piggybacking, dumpster diving, shoulder surfing, baiting** (e.g. dropped
  USB), **invoice/BEC fraud**.

> Portfolio link: [mirage](../../mirage) directly studies phishing/smishing/vishing and the
> behavioural constructs that make them work.

### 2.2 Malware

| Type | Defining trait |
|------|----------------|
| **Virus** | Needs a host file + user action to spread |
| **Worm** | Self-propagating over a network, no user action |
| **Trojan** | Disguised as legitimate software |
| **RAT** | Remote Access Trojan — attacker remote control |
| **Ransomware** | Encrypts data, demands payment |
| **Crypto-malware** | Cryptomining / crypto-ransomware |
| **Rootkit** | Hides at OS/kernel/firmware level; subverts the OS |
| **Bootkit** | Persists in the boot chain (defeated by verified/secure boot) |
| **Logic bomb** | Triggers on a condition (date, event) |
| **Keylogger / spyware** | Captures input / monitors activity |
| **Backdoor** | Bypasses normal auth |
| **Fileless** | Lives in memory / abuses LOLBins (e.g. PowerShell) |
| **Botnet** | Network of compromised hosts (bots) controlled via C2 |

> Portfolio link: [oracle](../../oracle)'s security-applications doc covers **malware
> visualisation** — classifying malware by rendering binaries as images.
> Verified boot in [nullbyte](../../nullbyte) defeats **bootkits/rootkits** at the boot stage.

### 2.3 Application / network attacks

- **Injection:** SQLi, command injection, LDAP/XML injection.
- **XSS** (stored/reflected/DOM) and **CSRF** — and the distinction: XSS abuses the user's trust in a
  site; CSRF abuses a site's trust in the user's browser.
- **Buffer overflow**, integer overflow, **race condition (TOCTOU)**, memory leaks, **DLL injection**.
- **Replay**, **MITM / on-path**, **session hijacking**, **SSL stripping**.
- **DoS / DDoS** (volumetric, protocol, application-layer); **amplification/reflection** (DNS, NTP).
- **DNS poisoning, ARP poisoning, MAC flooding, MAC/IP spoofing, rogue AP/evil twin, deauth**.
- **Pass-the-hash, pass-the-ticket, privilege escalation, password attacks** (brute force,
  dictionary, **spraying** — few passwords across many accounts, **credential stuffing** — reused
  breached creds, rainbow tables vs salting).
- **Zero-day** — no patch exists yet.

> Portfolio link: [spectre](../../spectre) demonstrated real findings — directory listing (CWE-548),
> version disclosure (CWE-200), missing security headers — and the GET-only ethical scope.

### 2.4 Cryptographic attacks

Birthday attack (hash collisions), collision, downgrade (e.g. forcing weak TLS), **rainbow tables**
(defeated by salting), brute force, known-plaintext. (More in [domain 6](6.0-cryptography-pki.md).)

## 3. Vulnerabilities and their sources

- **Misconfiguration / weak defaults** (the spectre findings), **unpatched systems**, **EOL/legacy**,
  **weak cipher suites**, **default credentials**, **open permissions**, **unsecured root accounts**.
- **Zero-day**, **improper input handling/error handling**, **resource exhaustion**.
- **Supply-chain** vulnerabilities (third-party libraries, vendors).
- **Architectural:** flat networks (no segmentation), single points of failure.

## 4. Vulnerability management

The lifecycle: **identify → assess/prioritise → remediate → verify → report**, repeated.

- **Scanning types:** **credentialed vs non-credentialed**, **agent vs network-based**, **intrusive
  vs non-intrusive**.
- **Scanner output:** **false positives** (flagged but not real — validate!), **false negatives**
  (missed — dangerous), severity often via **CVSS** (see below).
- **CVE** = catalogue of known vulnerabilities; **CVSS** = 0–10 severity score (base/temporal/
  environmental metrics). **CWE** = weakness *types* (e.g. CWE-548 directory listing).
- **Prioritise** by CVSS × asset criticality × exploitability (is it in **CISA KEV**? is there a
  public exploit?).
- **Remediation:** patch, configuration change, compensating control, or accept the risk.

> Portfolio link: [spectre](../../spectre) maps findings to **CWE** and estimates **CVSS**, and uses
> **CIS benchmarks** for hardening — the practical side of vulnerability management.

## 5. Penetration testing vs vulnerability scanning

| | Vuln scan | Pentest |
|---|----------|---------|
| Goal | Find known weaknesses | Exploit to prove impact |
| Automation | Mostly automated | Human-driven |
| Knowledge models | n/a | **Black / grey / white box** |
| Outcome | List of findings | Demonstrated attack paths |

Pentest concepts: **rules of engagement**, scope/authorisation, **pivoting**, **lateral movement**,
**persistence**, **privilege escalation**, **cleanup**, **bug bounty**, red/blue/**purple** team,
exercises with **white team** oversight.

> Portfolio link: [spectre](../../spectre) is a **grey-box** engagement; [gauntlet](../../gauntlet)
> documents CTF exploitation methodology.

## 6. Indicators of compromise (IoCs)

Unusual outbound traffic / beaconing, anomalous logins (impossible travel), new admin accounts,
unexpected scheduled tasks/cron, file-integrity changes, spikes in DNS/NXDOMAIN, disabled logging.
Tie observations to **MITRE ATT&CK** techniques for a common language.
