# 4.0 — Identity & Access Management

## 1. AAA and the core concepts

- **Identification** (claim an identity) → **Authentication** (prove it) → **Authorization** (what
  you may do) → **Accounting/Auditing** (log what you did). "**AAA**" = Authentication,
  Authorization, Accounting.
- **Identity vs authentication vs authorization** — a common exam trap: *who you claim to be* vs
  *proving it* vs *what you're allowed to do*.

## 2. Authentication factors

| Factor | "Something you…" | Examples |
|--------|------------------|----------|
| Knowledge | **know** | password, PIN, security questions |
| Possession | **have** | hardware token, smartcard, **FIDO2 key**, phone (TOTP/push) |
| Inherence | **are** | fingerprint, face, iris, voice (biometrics) |
| Location | **are (somewhere)** | GPS/geolocation, IP |
| Behaviour | **do** | typing rhythm, gait |

- **MFA** = two or more **different** factor *types* (password + SMS code = 2 factors; two passwords
  = still one type). **2FA** is MFA with exactly two.
- **Biometric metrics:** **FAR** (false accept — security risk), **FRR** (false reject — usability),
  **CER/EER** (crossover where FAR=FRR — lower is a better biometric).

> Portfolio link: [ironveil](../../ironveil)'s **FIDO2 hardware-key** LUKS unlock is a *possession*
> factor (with **touch = user presence**). Its
> [luks2-fido2-reference](../../ironveil/hardening/research/luks2-fido2-reference.md) notes that
> touch-only = single-factor *have*; adding a **clientPin** would make disk unlock true MFA
> (have + know). A precise, real example of the factor model.

## 3. Authentication protocols & technologies

| Protocol | Use | Notes |
|----------|-----|-------|
| **Kerberos** | Windows/AD auth | Ticket-based (TGT/TGS), KDC, time-sensitive (needs NTP); mutual auth; vulnerable to pass-the-ticket, golden ticket |
| **LDAP / LDAPS** | Directory queries | LDAPS = TLS-protected |
| **RADIUS** | AAA, network access (802.1X, VPN) | UDP; encrypts only the password |
| **TACACS+** | AAA (Cisco) | TCP; encrypts the whole payload; separates AAA |
| **SAML** | Web SSO (federation) | XML assertions; IdP ↔ SP |
| **OAuth 2.0** | **Authorization** (delegated access) | Tokens; NOT authentication by itself |
| **OpenID Connect** | **Authentication** layer on OAuth 2.0 | Adds the identity (ID token) |
| **802.1X / EAP** | Port-based network access | EAP-TLS (cert-based, strongest), PEAP |
| **CHAP / PAP / MS-CHAP** | Legacy auth | PAP cleartext (avoid); CHAP challenge-response |

- **OAuth vs OIDC trap:** OAuth = *authorization* (delegated access to resources); **OpenID Connect**
  = *authentication* built on OAuth. SAML = older web-SSO/federation alternative.
- **SSO** (one login → many services) — convenience + central control, but a single point of
  compromise → pair with MFA. **Federation** = trust across organisations/domains.

## 4. Access control models

| Model | Basis | Example |
|-------|-------|---------|
| **DAC** (Discretionary) | Owner decides | Typical file permissions |
| **MAC** (Mandatory) | System-enforced labels/clearances | Military/SELinux; subject clearance vs object classification |
| **RBAC** (Role-Based) | Permissions via roles/job function | Enterprise standard |
| **ABAC** (Attribute-Based) | Policy over attributes (user/resource/context) | Fine-grained, dynamic |
| **Rule-based** | Global rules (e.g. ACL/time-of-day) | Firewall ACLs |

- **Principle of least privilege** and **need to know**; **separation of duties**; **job rotation**;
  **mandatory vacation** (detect fraud); **dual control**.

> Portfolio link: [spectre](../../spectre) finding **F6** — the host user was in `sudo` with
> unrestricted execution: a **least-privilege** violation; the fix is RBAC-style scoped sudo.

## 5. Account management

- **Account types:** user, **privileged/admin**, service, guest, shared/generic (avoid — breaks
  accountability).
- **Privileged Access Management (PAM):** vaulting, just-in-time access, session recording,
  credential rotation.
- **Lifecycle:** **provisioning → review/recertification → de-provisioning** (offboarding is a
  classic gap — disable promptly).
- **Policies:** password length/complexity/history/age, **lockout**, **disablement vs deletion**,
  **time-of-day/location restrictions**, **impossible-travel** detection, conditional access.
- **Credential management:** password managers/vaults, **don't hardcode secrets**, key management,
  secrets in a KMS/HSM.

## 6. Identity proofing & federation extras

- **Identity proofing** (verify a person before issuing credentials), **attestation**,
  **just-in-time provisioning**, directory services (AD/LDAP), **certificate-based auth** (smartcards/
  PIV/CAC — ties to PKI in [domain 6](6.0-cryptography-pki.md)).

## 7. Passwordless, passkeys & phishing-resistant auth

The direction of travel away from passwords — increasingly tested and directly relevant to the
portfolio.

- **Why passwords fail:** reuse, phishing, breach replay, brute force/spraying — most incidents start
  with a credential.
- **Passwordless options:** **FIDO2/WebAuthn passkeys** (public-key, origin-bound), platform
  authenticators (Windows Hello, Touch/Face ID), hardware security keys, certificate/smartcard
  (PIV/CAC), magic links/OTP (weaker).
- **Phishing-resistant vs not:** **FIDO2 and PIV/smartcard are phishing-resistant** (the secret never
  leaves the device and signatures are origin-bound). **TOTP, SMS, and push are *not*** — they can be
  relayed, SIM-swapped, or **MFA-fatigue/prompt-bombed**.
- **Roaming vs platform authenticator:** roaming = a portable key (Nitrokey/YubiKey) usable across
  devices; platform = bound to one device's secure hardware.

> Portfolio link: [ironveil](../../ironveil)'s **Nitrokey FIDO2** is a roaming authenticator used here
> for *disk unlock* rather than web login — the same WebAuthn/CTAP2 primitive (public-key, device-held
> secret, user-presence touch) the exam describes for passwordless auth. The factor-model nuance
> (touch-only = *have* only; add a PIN for true MFA) is worked through in §2 above and in
> [luks2-fido2-reference](../../ironveil/hardening/research/luks2-fido2-reference.md).

## 8. Zero Trust identity & conditional access

- **Identity is the new perimeter:** in Zero Trust (domain 3 §1) every request is authenticated and
  authorised on identity + device posture + context — not network location.
- **Conditional access policies:** allow/deny/step-up based on user risk, device compliance,
  location/impossible-travel, app sensitivity (e.g. "require MFA + managed device for finance app").
- **Just-in-time (JIT) & least-privilege:** grant elevated rights only when needed, time-boxed,
  logged — PAM (§5) operationalises this.
- **Continuous authentication / risk-based auth:** re-evaluate trust mid-session on behavioural
  signals, not just at login.

> Portfolio link: [nullbyte](../../nullbyte)'s per-profile compartmentalisation is conditional access
> applied to a device — each operational identity is a separate trust boundary with its own
> credentials and policy, the practical shape of "identity as the perimeter."
