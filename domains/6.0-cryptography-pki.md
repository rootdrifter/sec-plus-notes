# 6.0 — Cryptography & PKI

## 1. Goals of cryptography (CIA + more)

- **Confidentiality** (encryption), **Integrity** (hashing/MAC), **Authentication** (signatures/MAC),
  **Non-repudiation** (digital signatures — can't deny you signed).
- **Obfuscation** (make harder to understand, not truly secret). **Diffusion/confusion** (cipher
  design properties). **Perfect forward secrecy (PFS)** — session keys not compromised even if a
  long-term key later is (ephemeral key exchange).

## 2. Symmetric vs asymmetric

| | Symmetric | Asymmetric |
|---|-----------|------------|
| Keys | **One shared secret** key | **Key pair** (public + private) |
| Speed | **Fast** — bulk data | Slow — small data / key exchange |
| Problem solved | Bulk encryption | **Key distribution**, signatures |
| Examples | **AES** (128/192/256), ChaCha20, 3DES (legacy), Blowfish/Twofish | **RSA**, **ECC/ECDSA/EdDSA**, **Diffie–Hellman**, ElGamal |
| Challenge | Securely sharing the key | Slower; needs PKI to trust public keys |

- **Hybrid (the real world):** asymmetric to *exchange/agree* a symmetric session key, then symmetric
  for the bulk data. **This is how TLS works** (and WireGuard, see below).
- **Block vs stream ciphers:** block (AES) operates on fixed blocks with **modes** (see §3); stream
  (ChaCha20, RC4-legacy) encrypts bit/byte-wise.
- **ECC** gives equivalent security to RSA at **much smaller key sizes** (e.g. 256-bit ECC ≈ 3072-bit
  RSA) — efficient for mobile/constrained devices.

> Portfolio link: **WireGuard** in [ironveil](../../ironveil) is a textbook hybrid:
> **Curve25519** (asymmetric, ECDH key agreement) → **ChaCha20-Poly1305** (symmetric AEAD for data),
> with **BLAKE2s** hashing. Its
> [adguard-wireguard-reference](../../ironveil/hardening/research/adguard-wireguard-reference.md) §3
> breaks down the Noise-framework handshake and PFS via ephemeral keys.

## 3. Block cipher modes & AEAD

- **ECB** — insecure (identical plaintext blocks → identical ciphertext; "the ECB penguin"). Avoid.
- **CBC** — chaining with an **IV**; needs padding; malleable without a MAC.
- **CTR** — turns a block cipher into a stream cipher.
- **GCM / AEAD** — **authenticated encryption with associated data**: confidentiality **+** integrity
  in one (AES-GCM, ChaCha20-Poly1305). Preferred modern choice.
- **XTS** — disk encryption mode (used by LUKS/BitLocker); resists certain block-manipulation attacks
  on storage.

> Portfolio link: [ironveil](../../ironveil)'s **LUKS2** uses **aes-xts-plain64** — XTS is the
> storage-encryption mode; see
> [luks2-fido2-reference](../../ironveil/hardening/research/luks2-fido2-reference.md) §1.

## 4. Hashing & integrity

- **Hash = one-way, fixed-length digest.** Provides **integrity**, not confidentiality.
- **Algorithms:** **SHA-2 (SHA-256/512)** good; **SHA-3** newer; **MD5 and SHA-1 are broken**
  (collisions) — don't use for security.
- **Collision** = two inputs, same hash (birthday attack). **Avalanche effect** = tiny input change →
  totally different hash.
- **HMAC** = hash + secret key → integrity **+** authenticity of a message.
- **Password storage:** never plain hash — use **salt** (defeats rainbow tables), **key-stretching /
  slow KDFs**: **bcrypt, scrypt, PBKDF2, Argon2**. **Pepper** = secret added in addition to salt.

> Portfolio link: [spectre](../../spectre) / [gauntlet](../../gauntlet) use **SHA-256** for evidence
> integrity. [ironveil](../../ironveil)'s LUKS2 uses **Argon2id** (memory-hard KDF) for passphrase
> keyslots — and HMAC appears below in FIDO2.

## 5. PKI — Public Key Infrastructure

The trust framework that lets you believe a public key belongs to whom it claims.

- **CA (Certificate Authority):** issues/signs certificates; the **root CA** is the trust anchor
  (kept offline), **intermediate/subordinate CAs** issue day-to-day → a **chain of trust** up to a
  trusted root.
- **RA (Registration Authority):** verifies identity before the CA issues.
- **Digital certificate (X.509):** binds an identity to a public key; signed by the CA. Fields:
  subject, issuer, validity, public key, **SAN** (Subject Alternative Names), serial.
- **CSR (Certificate Signing Request):** what you send the CA (contains your public key + identity).
- **Revocation:** **CRL** (Certificate Revocation List — periodic) vs **OCSP** (Online Certificate
  Status Protocol — real-time) and **OCSP stapling** (server presents a recent signed status).
- **Trust models:** hierarchical (CA tree), **web of trust** (PGP), cross-certification, **pinning**.
- **Certificate types:** DV / OV / **EV**, wildcard (`*.example.com`), SAN/multi-domain,
  **self-signed** (no external trust), code-signing, client/machine certs, **root**.
- **Key concepts:** **key escrow** (third party holds a copy — recovery vs risk), **M-of-N control**,
  key recovery agent, certificate/key lifecycle (enrol → issue → renew → revoke → expire).

> Portfolio link: **certificate pinning** is exactly what [ironveil](../../ironveil)'s dracut-sshd
> does with the **initramfs SSH host key** (pinned on the GrapheneOS client to stop MITM) — see
> [dracut-sshd-reference](../../ironveil/hardening/research/dracut-sshd-reference.md). [spectre](../../spectre)
> evaluated a target's **self-signed** TLS cert (no revocation chain) as a finding limitation.

## 6. Digital signatures & TLS handshake

- **Digital signature:** hash the message → encrypt the hash with the **sender's private key**.
  Recipient verifies with the sender's **public key** → integrity + authentication + **non-repudiation**.
  (Encryption uses the *recipient's* keys; signing uses the *sender's* — a classic exam trap.)
- **TLS handshake (concept):** negotiate cipher suite → authenticate server via its **certificate**
  (chain to a trusted CA) → **key exchange** (ECDHE for PFS) to agree a symmetric session key →
  symmetric AEAD (AES-GCM/ChaCha20-Poly1305) for the bulk session. **TLS 1.3** removed legacy/weak
  options and is faster (1-RTT).

## 7. Authentication, FIDO2 & hardware crypto

- **FIDO2 / WebAuthn + CTAP:** phishing-resistant authentication using **public-key crypto** — the
  private key never leaves the authenticator; the relying party stores only the public key. Defeats
  credential phishing/replay because signatures are origin-bound and challenge-based.
- **hmac-secret (CTAP2 extension):** lets a FIDO2 authenticator compute **HMAC** over a salt with a
  per-credential secret held *inside the device* — the basis for using a security key to derive a
  disk-encryption secret.
- **Hardware:** **TPM** (platform integrity + sealed keys), **HSM** (high-assurance key storage/ops),
  **secure element / smartcard / PIV-CAC**, **hardware security keys** (Nitrokey/YubiKey).

> Portfolio link: this domain is *built* in [ironveil](../../ironveil). The **Nitrokey 3A NFC**
> uses FIDO2 **hmac-secret** to derive the LUKS2 keyslot — high-entropy, device-held, never typed.
> The [luks2-fido2-reference](../../ironveil/hardening/research/luks2-fido2-reference.md) explains
> the libfido2/CTAP2 flow and the **touch-only (presence, no PIN)** trade-off — i.e. *something you
> have* without *something you know*. The single clearest theory-to-practice link in this portfolio.

## 8. Cryptographic attacks & limitations

- **Brute force, dictionary, rainbow tables** (→ salt), **birthday/collision** (→ strong hashes),
  **downgrade** (→ disable weak protocols), **known/chosen plaintext**, **side-channel** (timing,
  power), **replay** (→ nonces/timestamps).
- **Limitations:** key management is the hard part; **entropy/randomness** matters (weak RNG = weak
  keys); resource constraints (lightweight crypto for IoT); **crypto-agility** (ability to swap
  algorithms). **Quantum** threat — Shor's algorithm threatens RSA/ECC → **post-quantum cryptography**
  (PQC) is emerging; **harvest-now-decrypt-later** is the present concern.

> Portfolio link: [ironveil](../../ironveil) notes WireGuard's optional **pre-shared key** as a
> post-quantum hardening factor (the `psk` in `Noise_IKpsk2`) — crypto-agility in practice.

## 9. Key & secrets management lifecycle

Key management — not algorithm choice — is where real systems fail.

- **Lifecycle:** **generate → distribute → store → use → rotate → revoke/expire → destroy**, with an
  audit trail at each step. Strong **entropy** at generation is critical (weak RNG = weak keys).
- **Storage:** never in source/config — use an **HSM**, **KMS**, secure element/TPM, or a secrets
  vault. **Hardware-backed keys never leave the device.**
- **Rotation & revocation:** rotate on schedule and on compromise; revoke promptly (CRL/OCSP for
  certs). Plan for **key recovery** without weakening security.
- **Escrow vs recovery:** **key escrow** (a third party holds a copy — enables recovery but adds a
  trust/exposure risk) vs **M-of-N control** (split-knowledge, e.g. 3-of-5 must combine) and a
  **recovery agent**. **Separation of duties** over key custody.
- **Backups:** an encrypted backup is only as recoverable as its key — protect and physically
  separate recovery material.

> Portfolio link: [ironveil](../../ironveil) practises this end-to-end: a **backup hardware key + LUKS
> recovery material kept in a Pelican 1200 case**, physically separated from the running machine
> (split custody), and a documented **ed25519 key-rotation** procedure for the dracut-sshd initramfs
> host key. The FIDO2 keyslot secret is **device-held and never typed** (§7) — textbook
> hardware-backed key storage.

## 10. Crypto use-cases, constraints & selection

The exam asks you to *match* a primitive to a requirement under real constraints.

- **Use-case → primitive:** confidentiality → symmetric AEAD; integrity → hash/HMAC; authenticity +
  non-repudiation → digital signature; key exchange → ECDHE; data at rest → XTS/FDE; data in transit →
  TLS/IPsec/WireGuard; password storage → slow KDF (Argon2).
- **Constraints that drive choice:** **resource/low-power** (IoT → lightweight/ECC over RSA),
  **low-latency** (fast symmetric, session resumption), **high-resiliency/longevity** (larger keys,
  crypto-agility), **entropy** availability, regulatory (FIPS-validated modules).
- **Trade-offs:** key size vs speed; PFS (ephemeral keys) vs session-resumption cost; hardware-backed
  assurance vs portability.

> Portfolio link: [ironveil](../../ironveil)'s stack is a constraint-driven selection: **ChaCha20-
> Poly1305** in WireGuard (fast, constant-time, mobile-friendly where AES-NI may be absent),
> **AES-XTS** for LUKS2 storage, **Argon2id** (memory-hard) for the human-passphrase keyslot — each
> primitive matched to its job and platform.

### Quick self-test
- Symmetric vs asymmetric — which is fast/bulk, which solves key distribution?
- Signing: whose key encrypts the hash — sender's or recipient's **private**?
- CRL vs OCSP — periodic list vs real-time check?
- Why salt passwords, and why a *slow* KDF (Argon2/bcrypt) over plain SHA-256?
- AEAD (GCM) — what two properties in one?
- Why is FIDO2 **phishing-resistant** where TOTP/SMS is not?
- Name the key-management lifecycle stages from generation to destruction.
- Key escrow vs M-of-N control — recovery benefit vs the trust risk of each.
- Why might a constrained IoT device prefer **ECC** over RSA, and **ChaCha20** over AES?
