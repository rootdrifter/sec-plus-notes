# 3.0 — Architecture & Design

## 1. Secure network architecture

- **Segmentation:** **VLANs**, subnets, **DMZ/screened subnet** (public-facing servers isolated from
  the internal LAN), **east-west vs north-south** traffic, microsegmentation.
- **Zero Trust:** "never trust, always verify" — no implicit trust by network location; verify every
  request (identity + device posture + context). Contrast with the old "hard perimeter, soft inside".
- **Defence in depth:** layered controls so no single failure is fatal.
- **Other patterns:** air-gapped networks, **jump box/bastion** for admin, honeypots/honeynets (decoy
  to detect/study attackers), network **TAPs**, port mirroring/SPAN, **NAC** posture checks, **802.1X**.
- **Designs:** **IPv4/IPv6**, NAT/PAT, **software-defined networking (SDN)**, secure protocols
  (domain 2 §3).

> Portfolio link: [nullbyte](../../nullbyte)'s **nine-profile compartmentalisation** is segmentation
> applied to a device — each profile is a boundary, the **Satan** pentest profile isolated from
> **Mammon** financial. [ironveil](../../ironveil) routes all egress through one tunnel with a
> kill-switch (a defence-in-depth network control).

## 2. Defence-in-depth control types

Two orthogonal classifications the exam loves to combine:

**By function:**
- **Preventive** (firewall, ACL, MFA), **Detective** (IDS, SIEM, logs), **Corrective** (backups,
  patching), **Deterrent** (warning banners, fences), **Compensating** (alternative when primary
  isn't feasible), **Recovery** (DR), **Directive** (policy).

**By nature:**
- **Technical/logical** (encryption, firewalls), **Administrative/managerial** (policies, training),
  **Physical** (locks, guards, mantraps, bollards).

## 3. Secure systems & host design

- **Hardening:** disable unnecessary services/ports, remove default accounts/creds, patch, apply
  **secure baselines / benchmarks (CIS, STIG)**, least functionality.
- **Boot integrity:** **Secure Boot / Measured Boot / UEFI**, **TPM** (platform integrity + key
  storage), **HSM** (dedicated crypto hardware), **trusted boot chain / attestation**.
- **Endpoint:** FDE, host firewall, EDR, application allow-listing, FIM.
- **Trusted execution / hardware roots of trust:** TEE, secure enclave/secure element.

> Portfolio link: [spectre](../../spectre) hardened PostgreSQL to **CIS Level 1** and recommended
> disabling unneeded services (`snapd`, `ModemManager`). [nullbyte](../../nullbyte)'s **verified
> boot + Titan-class secure element** is boot integrity + a hardware root of trust in practice; its
> [grapheneos-security-model](../../nullbyte/architecture/research/grapheneos-security-model.md) maps
> the chain.

## 4. Virtualisation & cloud security

### 4.1 Virtualisation
- **Hypervisor Type 1** (bare-metal, e.g. ESXi) vs **Type 2** (hosted, e.g. VirtualBox).
- **VM escape** (break out to host — the key risk), **VM sprawl**, **snapshots** for rollback.
- **Containers** (Docker/Kubernetes): lighter than VMs, share the host kernel → weaker isolation than
  VMs; image vulnerabilities, orchestration security.

> Portfolio link: [spectre](../../spectre) was built in **VirtualBox (Type 2)** with snapshots for
> clean rollback at each stage.

### 4.2 Cloud
- **Models:** **IaaS / PaaS / SaaS** (and the **shared responsibility model** — who secures what at
  each tier; the customer always owns *data* and *access*).
- **Deployment:** public / private / hybrid / community.
- **Cloud controls:** **CASB** (visibility/policy between users and cloud apps), **SWG** (secure web
  gateway), security groups/NSGs, IAM roles, KMS, **misconfiguration** (the #1 cloud breach cause —
  open S3 buckets, over-permissive IAM), **CSPM**.
- Concepts: serverless/functions, **VPC**, transit gateways, edge/fog computing.

## 5. Resilience & redundancy

- **RAID** (0 stripe/no redundancy, 1 mirror, 5 parity, 6 dual parity, 10 mirror+stripe).
- **High availability:** clustering, load balancing, **active/active vs active/passive**, **N+1**.
- **Geographic redundancy**, **failover**, **UPS/generators**, **NIC teaming**, multipath.
- **Backups:** **full / incremental / differential** (know the restore-time/backup-size trade-offs),
  **3-2-1 rule**, offsite, **immutable/WORM** backups vs ransomware, snapshots, journaling.
- **Non-persistence:** revert to known state, live boot, **snapshots**, master images.

## 6. Physical & environmental

- **Perimeter:** fencing, bollards, lighting, **mantrap/access control vestibule**, signage, guards,
  CCTV, sensors.
- **Access:** badge/proximity cards, biometric readers, locks (cipher, biometric), **mantraps** to
  stop tailgating, **faraday cage** (RF shielding).
- **Environmental:** HVAC, **hot/cold aisle** containment, fire suppression (clean-agent vs water),
  humidity/temperature monitoring, EMI/EMP shielding.

> Portfolio link: [ironveil](../../ironveil) stores its backup hardware key + recovery material in a
> **Pelican 1200** protective case — physical custody control for recovery material, separated from
> the running machine.

## 7. Data security & privacy by design

- **Data states:** **at rest** (FDE/file encryption), **in transit** (TLS/VPN), **in use** (TEE,
  homomorphic encryption — emerging).
- **Data classification:** public / internal / confidential / restricted; **PII / PHI / SPI**.
- **Privacy techniques:** tokenization, **data masking**, anonymisation/pseudonymisation, **hashing**,
  **DLP**.
- **Privacy/secure-by-design** principles; data minimisation; data sovereignty/residency.

> Portfolio link: [ironveil](../../ironveil) = data **at rest** (LUKS2) + **in transit**
> (WireGuard). [mirage](../../mirage) used **synthetic (CVAE) vishing data** specifically to avoid
> privacy risk — privacy by design in research.
